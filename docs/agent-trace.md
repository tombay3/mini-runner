# Agent Trace Store

## Summary
`agent-traces.json` is the diagnostic store for the V2 Lode Runner candidate agent. It records recent backend planning steps, compact state summaries, generated candidate summaries, the selected candidate, validation, stall supervision, and model metadata.

Replayable demos live in [__data1/recordings.json](../__data1/recordings.json). Agent recordings reference traces with `traceId`.

## Storage And Retention
The Flask backend stores traces at [__data1/agent-traces.json](../__data1/agent-traces.json). The current store version is `1`.

Trace retention is intentionally bounded:

- `runs` keeps up to the 10 newest trace runs.
- When an eleventh run is added, the oldest retained run is deleted.
- Old recordings may reference traces that have since been pruned.

## API
Trace-related routes are implemented in [app.py](../app.py).

`POST /api/agent/next-action`

Creates/appends one step in a trace run and returns the normal agent action response:

```json
{
  "action": { "keyCode": 39, "ticks": 8, "reason": "..." },
  "planner": {},
  "traceId": "...",
  "stepCount": 2,
  "candidateId": "...",
  "candidate": {},
  "candidates": [],
  "validation": {}
}
```

`GET /api/agent/traces/<trace_id>`

Returns one retained trace run, or `404` if that run has been pruned.

`GET /api/agent/runs/<playData>/<level>`

Returns latest run metadata plus the saved recording for that context if either exists.

## Store Shape
Top-level shape:

```json
{
  "version": 1,
  "updatedAt": "2026-05-28T04:05:50.460Z",
  "runs": {
    "<traceId>": {}
  }
}
```

Each run contains:

- `id`: trace/run id.
- `createdAt` / `updatedAt`: first and latest trace timestamps.
- `playData` / `level`: game context.
- `model`: compact model summary from the latest planner.
- `stepCount`: number of persisted backend planning steps.
- `latestAction`: action returned by the latest step.
- `steps`: ordered lean step traces.

`stepCount` is the total number of backend decisions. Step indexes are zero-based, so `stepCount = 2` means step indexes `0` and `1`.

## Step Shape
Each step is generated by [agent/traces.py](../agent/traces.py), then [app.py](../app.py) adds `stepIndex`, `playData`, and `level`.

Lean V2 step fields:

- `createdAt`: step timestamp.
- `stepIndex`: zero-based index inside the run.
- `playData` / `level`: game context.
- `state`: normalized state summary derived from V2 candidate analysis.
- `candidates`: compact generated candidate summaries.
- `selectedCandidateId` / `selectedCandidateKind`: final candidate selected after validation/retry/fallback.
- `validation`: selected-candidate validation and fallback metadata.
- `action`: final translated `{ keyCode, ticks, reason }` sent to the browser.
- `planner`: model metadata and candidate-selection outcome.
- `stallSupervisor`: deterministic stall/loop supervisor summary.
- `historyTail`: recent browser-executed action history.

Removed legacy/debug-heavy fields:

- `requestedModel`
- `selectedModel`
- `snapshot`
- full `analysis`
- full `selectedCandidate`
- duplicate `guardrail`
- `finalMessage`
- `intermediateMessages`
- `response`
- `runMode`
- `aisuiteProvider` / `aisuiteModel`

## State Summary
`state` is the only persisted game-state summary for a step. It replaces the older `snapshot` plus full `analysis` duplication.

Typical fields:

- `gameState`, `tick`, `godMode`
- `goldCount`, `goldComplete`, `gold`
- `runner`, `guards`
- `nearestGold`, `primaryProgressTarget`
- `rowLadders`
- `risk`
- `movement`, `dig`, `ladder`, `routeAccess`
- `stallReport`

The trace does not store the full raw browser snapshot.

## Raw Model I/O Debug Log
Raw prompts and provider message content are intentionally kept out of `agent-traces.json`.

For prompt/model debugging, enable a separate rotating local log:

```sh
AGENT_DEBUG_LOG=1 python app.py
python app.py --debug
```

When enabled:

- raw model I/O is written to `__data1/agent-debug.log`
- only the latest 10 model turns are retained
- app logging is raised to `DEBUG`
- stdout still does not include raw prompts or raw model output

Each debug block includes the trace id, model, retry flag, raw `build_agent_prompt()` output, final message content, optional provider `reasoning_content`, parse error, and selected candidate id.

## Candidate Data
The V2 agent is candidate-centric. The model chooses a generated `candidateId`; it does not invent raw key codes.

Each persisted candidate summary includes:

- `id`
- `kind`
- `score`
- `baseScore`
- `stallBlocked`
- `stallRecovery`
- `target`
- `firstAction`
- `goal`
- `reason`

The live API response may still return the full selected candidate and candidate list for immediate debugging, but persisted traces keep only summaries.

## Planner Metadata
`planner` is logical model metadata and selection outcome:

- `modelProfile`: selected profile, such as `openai`, `minimax`, `gemini`, or `explicit`.
- `provider`: logical provider.
- `model`: logical `provider:model` string.
- `modelSource`: request/config source.
- `mode`: currently `candidate-selection`.
- `generatedAt`: provider response timestamp when available.
- `responseId`: provider response id when available.
- `fallbackUsed` / `fallbackReason`
- `candidateCount`

Transport-specific fields are intentionally omitted. For example, Gemini may use an OpenAI-compatible transport internally, but the trace records the logical provider/model as `gemini`.

## Validation And Stall Metadata
`validation` explains how the candidate was accepted or changed:

- `knownCandidate`
- `requestedCandidateId`
- `selectedCandidateId`
- `actionValid`
- `fallbackUsed`
- `fallbackReason`
- `stallBlocked`
- `stallBlockReason`
- `stallReportType`
- `stallSeverity`

`stallSupervisor` stores the high-level stall/loop decision for the step. Current stall types may include:

- `horizontal_oscillation`
- `vertical_ladder_oscillation`
- `same_candidate_no_progress`
- `same_tile_no_progress`
- `route_access_loop`
- `exit_ladder_loop`
- `wait_loop`

## Reading A Trace
For quick inspection:

1. Find the newest run in `runs` for the target `playData` / `level`.
2. Open `runs[traceId].stepCount`.
3. Inspect `runs[traceId].latestAction`.
4. Inspect the last item in `runs[traceId].steps`.
5. Compare `state.runner`, `state.gold`, `state.primaryProgressTarget`, `state.stallReport`, `candidates`, `selectedCandidateId`, `validation`, and `action`.

Useful one-liner:

```sh
python3 - <<'PY'
import json
data = json.load(open("__data1/agent-traces.json"))
runs = data["runs"]
latest, run = max(
    (
        (trace_id, run)
        for trace_id, run in runs.items()
        if str(run.get("playData")) == "1" and str(run.get("level")) == "1"
    ),
    key=lambda item: item[1].get("updatedAt") or item[1].get("createdAt") or "",
)
print(latest, run["stepCount"], run.get("latestAction"))
PY
```

## Trace Versus Logs
Traces carry detailed debugging context. Logs, documented in [docs/logging.md](./logging.md), stay operational and compact.

Use traces for:

- candidate lists
- compact state summaries
- stall reports
- validation/fallback outcomes
- model metadata

Use logs for:

- selected-step summaries
- recording saved events
- config/model errors
- trace/recording persistence errors

## Current Boundaries
The trace store is local diagnostic data, not a long-term analytics database.

- Retention is capped at 10 runs.
- There is no browser-side trace viewer.
- There is no full raw snapshot persistence.
- V1/tool-heavy trace transcript fields are not supported.
- Runtime benchmark trace fields are not supported.
